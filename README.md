# 絶対負けない後出しじゃんけん番所

## 作ったもの(1行)

「じゃん、けん、ぽん!」の掛け声演出つきで、Webカメラの手をブラウザ内ML(MediaPipe Hands)で認識し、人間の視覚反応(約200ms)より速く勝つ手を後出しするじゃんけんマシン。勝率100%(SLA保証)。

**禁じ手ルール:** 当機が手を出した後にあなたが手を変えると「後出し検出→貴殿の反則負け」。自分は後出しするくせに、人の後出しは1フレーム単位で検出して許さない。

## 遊ぶ

**▶ https://k-iijima.github.io/atodashi-janken/**

1. ページを開いてカメラを許可する(映像はブラウザ内で処理され、外部には一切送信されない)
2. 手をカメラに向けると「じゃん、けん、ぽん!」が自動で始まる
3. 「じゃん、けん」で拳を振り、「ぽん!」で ✊✌️✋ のどれかを出す
4. 当機が人間には知覚できない速さで後出しして勝ち、その瞬間が**判定写真**として画面に残る
5. カウントダウン(3・2・1)ののち、手を出し続けている限り次の勝負が始まる

出すのが少し遅れて判定とズレても、直後(0.5秒以内)なら当機が判定をやり直してくれる。ただし当機は訂正を認めず、「**最初からこれを出しておりましたが、何か?**」と手をしれっと差し替え、撮り直した証拠写真に「**判定写真(無加工)**」の札を貼る。それより遅い手替えは当機の手を見てから変えたとみなし、「後出し!」として**貴殿の反則負け**(証拠の**犯行写真**つき)。手を引っ込めるだけならセーフ。場内掲示板には対戦数・勝率(常に100%)・後出し時間(ms)・貴殿の反則回数が表示される。

カメラが複数ある場合は、挑戦者側の下に「撮影機」プルダウンが現れて切り替えられる(選択は記憶される)。

## ローカルで動かす(開発用)

ES module 構成のため `file://` では動かない。ローカルサーバを1本立てる:

```
python -m http.server 8000
# → http://localhost:8000
```

main へ push すると GitHub Actions([.github/workflows/deploy.yml](.github/workflows/deploy.yml))が自動で Pages に反映する。

## デザイン方針(AIっぽさの脱却)

参考:[AIっぽいデザインの脱却ガイド](https://nextage-tech.com/blog/2026/06/08/post-7030/)

AI生成デザインの典型(紫グラデ・過剰な光彩・丸カード・無個性フォント)を全部やめて、**昭和の駄菓子屋ゲーセン×漫画**に振り切った:

- 生成り紙の背景+SVG `feTurbulence` によるノイズテクスチャ(画像ファイル不要)
- 光彩(glow)の代わりに**漫画の集中線**(`repeating-conic-gradient` を放射マスクで抜く)
- 丸カードの代わりに角ばった黒枠+ぼかさない「ベタ落ち影」+わずかな傾き(手貼り感)
- フォントは筆文字(Yuji Syuku)・レトロ見出し(RocknRoll One)・電光掲示板(DotGothic16)の混植
- 勝敗は朱色の**ハンコ**(「勝」「反則負け」)をドンと捺す

## 技術的面白ポイント(3行)

- オンデバイスML:MediaPipe Hands(WASM)でブラウザ完結。API不要・映像の外部送信なし
- 指の伸展判定は関節角度ではなく「手首からの距離比(tip vs pip)」。手の回転・傾きに強い
- 誤認識対策のNフレーム一致(デバウンス)と「後出し時間」のトレードオフが、掲示板の ms 表示でそのまま可視化される

## ハマった所(1つ)

グー→パーの遷移中に一瞬「チョキ」と誤認識され、当機が手を出し直す(**後出しの後出し**)。安定化フレーム数を増やすと直るが、今度は反応が遅れて人間に「後出し感」がバレる。現状は3フレーム(約100ms)で妥協。

反則検出も同じトレードオフの裏返しで、勝負判定より厳しめの5フレーム一致にしないと、手を引っ込める途中の誤認識で無実の人が反則負けになる。それでも勝負後に手を下ろすだけで冤罪が多発したため、「反則監視は当機が出した直後だけ」+「判定の瞬間を写真で凍結して表示」に落ち着いた。

一番しぶとかったのは**境界ギリギリ問題**:「じゃん、けん」で振っている拳(グー)が「ぽん!」直後に先に確定してしまい、本当の手が一瞬遅れて届くと毎回「手替え=後出し」にされる。対策は二段構えで、(1) ぽん!の瞬間から出しっぱなしの手は確定を約0.4秒粘る(振りの残りを即決しない)、(2) それでも確定直後0.5秒以内に届いた別の手は反則ではなく判定をやり直す。人間が当機の手を見てから変えるには最短0.4〜0.6秒かかるので、それより速い手替えは「元々出すつもりだった手」と推定できる、というのが猶予の根拠。

するとこの粘りのせいで、今度は**グーを普通に出した人だけ毎回0.4秒待たされる**(振りの拳とグーは形が同じなので区別できない)。決め手は形ではなく**動き**:振りの途中の拳は動き続けるが、出し終えた手は止まる。手首ランドマークの1フレームあたり移動量を測り、静止が4フレーム続いたら「出した手」とみなして粘りを免除した。振りの折り返し点でも手首は1〜2フレーム止まって見えるので、4フレーム連続という条件がそれとの区別になっている。

ところがこの救済で今度は**当機の後出し感**が強くなった(判定を粘る=出すのが遅れる、堂々と「訂正」と表示する)。対策は認知の側で、(1) ぽん!と同時に「投げている最中」の見た目(形が読めない高速の振り+ブラー)を出し、数百ms後の確定を「振り下ろしの着地」に見せる、(2) 判定のやり直しは一切騒がず、「最初からこれを出しておりましたが、何か?」と主張して証拠写真に「無加工」の札を貼る。イカサマは隠してこそ、という学び。

あと、**十数戦すると突然手を検出しなくなる**ことがあった。正体は認識ループの突然死:`await hands.send()` が一度でも例外を投げる(MediaPipeのWASMは長時間動かすと稀にやる)と、`requestAnimationFrame` の再登録に届かずループが静かに終わる。映像は流れ続けるので固まったようには見えない、という嫌なやつ。対策は (1) send を try/catch してループは何があっても続ける、(2) 失敗が3連続したら認識エンジン(Handsインスタンス)ごと作り直す、(3) 「失敗もせず結果も返さない」沈黙への保険として、結果が10秒途絶えたら作り直すウォッチドッグ。長時間動かすデモの基本装備という学び。
